##前端代码审查

###目的
   * 避免低级bug
   * 统一代码习惯
   * 保证线上代码质量

####审查清单（Code Review）

1、常规项
 
 ```
代码是否能够正常运行？
代码有没有实现预期的功能？逻辑是否正确？
是否遵循团队要求的代码规范？
是否有被注释掉的代码？
是否有可以被库函数替代的代码？
是否有冗余代码没有注释掉或者删掉？例如：删除或者注释 console.log 避免低端 IE 报错等。
CSS 属性是否有拼写错误？
HTML 标签是否书写正确，是否嵌套正确？
代码是否经过 ESLint 工具校验？
变量名是否简单易懂？是否拼写正确？是否有可以被替换的全局变量？
 ```
 
2、注释说明
 
```
是否有注释，并且描述了代码的意图？
是否在重要功能附近添加合适的注释？
所有的函数都有注释吗？
注释是否包含了相关功能说明？
对非常规行为和边界情况处理是否有描述？
换位思考，你能根据你的注释推断出下面代码的功能吗？
代码里是否还存在 TODO、 debug？
代码是否可以删掉或者完善功能？
是否有未完成的代码？如果有，是不是应该移除，或者用合适的标记进行标记比如’TODO’？
可能产生意外情况的地方是否留下说明？

```

3、代码质量

```
所有的代码是否逻辑清晰、简单易懂？
循环是否设置了长度和正确的终止条件？
关键功能是否还有优化的空间？
代码是否尽可能的模块化、组件化？
是否将核心功能尽可能独立，从而避免其他功能出现问题影响到核心功能？
```

4、安全与性能

```
所有的数据输入是否都进行了检查（检测正确的类型，长度，格式和范围）并且进行了编码？
在哪里使用了第三方工具，返回的错误是否被捕获？
在哪里使用第三方cdn文件或者图片？
输出的值是否进行了检查？
无效的参数值是否能够处理？
JS 代码是否尽量放在底部？
CSS 代码是否尽量放在了顶部？
是否部署 CDN 或者开启了缓存功能？
对静态资源图片、第三方插件的特殊处理？
使用预加载、懒加载、异步加载等技术手段提高性能。
DOM 操作有关的代码时，是否对 DOM 不存在或者被人为修改的情况做处理？
获取数据和信息时，是否对类型做过处理和转换并设置为空时的默认值？比如：var num = parseInt(Str);

```

####形式与流程

1、人员

```
组内前端人员有2人及以上人数，组内前端人员之间进行Code Review(正常情况下)；
组内人员不足2人，报备前端技术组长，由前端技术组长进行协调Code Review人员；
新人试用期间，由导师和组里前端人员共同Code Review新人代码。
```

2、时间与周期

```
1、当天开发的代码当天提交，最晚于下班前git代码到自己分支；
2、按照2天一次进行Code Review；
3、项目上线前进行前端技术组的 Code Review；
```
```
开发人员按照组件、模块、功能进行开发；
开发完毕后git到自己的分支并push到远端；
未完成的代码提交git到本地分支。
```
```
可根据项目开发强度，负责人可适当调整Code Review时间；
新人试用期内，每天下班前，导师和组里前端人员共同Code Review代码；
```

3、流程

```
（1）开发人员按照组件、模块、功能进行提交；
（2）Code Review前，提前阅读对方git到远端的代码；
（3）Code Review时，陈述在对方代码中发现的问题；
（4）参与Code Review的人员可就问题提出疑问，也可提出自己发现的问题；
（5）对于存在疑问的问题，共同协商确定；
（6）开发人员开发中，优先处理提出需要修改的问题；
（7）处理完毕后，向提出修改问题的人员进行核实；
（8）核实完毕后，不得随意更改原码，若必须更改，需提前告知相应的人员。

```
